Reward & Bounty
Kuncie is always committed to protecting customers in the use of our platform and in maintaining the security of user information is part of our main priority at Kuncie. Kuncie is pleased with the contributions of external security researchers and looks forward to recognizing their invaluable contribution to the security of all Kuncie users.If you (Security Researcher) comply with the policies below when reporting a security issue to Kuncie, we (Kuncie) will not file a lawsuit or law enforcement investigation against you in response to your report.
‍
Kuncie will be offering these external security researchers a prize for their invaluable contribution to improving security at Kuncie. Please send valid vulnerability details to support@kuncie.com
  1. Be the first one to report a specific vulnerability. Duplicate report is not eligible for bounty reward.
  2. Include details and verifiable proof of concept (e.g. screenshot, video, script). If our team cannot reproduce or verify an issue, a bounty cannot be awarded.
  3. Reporter eligible for bounty after Kuncie team decides to fix the bug.
  4. Do not share the vulnerability information to any party outside Kuncie without permission.
  5. Reporter of vulnerability is required to use their own accounts when performing testing, DO NOT attempt to view or tamper any data belonging to others.
  6. Reporter of vulnerability is prohibited to disturb, change, add, or delete any data or configuration inside Kuncie systems, targeting other users or compromising the reputation of Kuncie.
  7. Reporter of vulnerability is prohibited to take advantage of any vulnerability as a pivot or chain to find another vulnerability inside Kuncie systems.
  8. DO NOT perform DDoS or DoS attack to Kuncie systems.
  9. Kuncie reserves the right to cancel or modify this program at any time without prior announcement.

Application In Scope

  4. others related Kuncie environment

Vulnerability Scope

  1. SQL Injection 
  2. Cross-site Scripting (XSS) 
  3. Significant Authentication Bypass 
  4. Access Control Issues (Insecure Direct Object Reference issues, etc) 
  5. Cross-site Request Forgery in Critical Action 
  6. Information disclosure of Sensitive Information 
  7. Server-Side Request Forgery (SSRF) 
  8. Server-side Remote Code Execution (RCE) 
  9. XML External Entity Attacks (XXE) 
  10. Exposed Administrative Panels that don't require login credentials 
  11. Directory Traversal Issues 
  12. Local File Disclosure (LFD) 
  13. Server Side Template Injection (SSTI)

Out of Scope

  1. Testing environmentx

Out of Scope Properties

  1. 3rd Party Apps (Microsite, Wordpress, CMS, Blog, Even Inside kuncie.com/* ,etc. ) 
  2. 3rd Party Plugins

Out of Scope Vulnerability

  1. Self-XSS (we require evidence on how the XSS can be used to attack another Kuncie user).
  2. We will accept reports of XSS on Out of Scope Properties but will not reward for them.
  3. XSS issues that affect only outdated browsers.
  4. Reports that state that software is out of date/vulnerable without a proof of concept.
  5. Password, email and account policies, such as email id verification, reset link expiration, password complexity.
  6. Missing security headers which do not lead directly to a vulnerability.
  7. Missing best practices (we require evidence of a security vulnerability).
  8. Host header injections unless you can show how they can lead to stealing user data.
  9. Reports of spam (i.e., any report involving ability to send emails & SMS without rate limits).
  10. Stack traces that disclose information.
  11. CSV injection.
  12. Highly speculative reports about theoretical damage. Be concrete.
  13. Vulnerabilities as reported by automated tools without additional analysis as to how they're an issue.
  14. Reports of insecure SSL/TLS ciphers (unless you have a working proof of concept, and not just a report from a scanner).
  15. Reports from automated web vulnerability scanners (Acunetix, Vega, etc.) that have not been validated.
  16. Social Engineering (Phishing,Fraud, etc.).
  17. Denial of Service Attacks.
  18. Reflected File Download (RFD).
  19. window.opener (tabnabbing), related issues.
  20. Physical or social engineering attempts (this includes phishing attacks against Kuncie employees).
  21. Content injection issues.
  22. Most Brute Forcing issues
  23. Cross-site Request Forgery (CSRF) with minimal security implications (Logout CSRF, etc.)
  24. Missing autocomplete attributes.
  25. Phishing risk via unicode/punycode or RTLO issues.
  26. Being able to upload files with wrong extension in chooser.
  27. Missing cookie flags on non-security-sensitive cookies.
  28. Issues that require physical access to a victim’s computer.
  29. Missing security headers that do not present an immediate security vulnerability.
  30. Missing HTTP security headers, specifically, Example : Strict-Transport-Security, X-Frame-Options, X-XSS-Protection, X-Content-Type-Options, Content-Security-Policy, X-Content-Security-Policy, X-WebKit-CSP, Content-Security-Policy-Report-Only
  31. Fraud issues (please see the below section elaborating on this).
  32. SSL/TLS scan reports (this means output from sites such as SSL Labs).
  33. Banner grabbing issues (figuring out what web server we use, etc.).
  34. Open ports without an accompanying proof-of-concept demonstrating vulnerability.
  35. Recently disclosed 0day vulnerabilities. We need time to patch our systems, please give us 1 month before reporting these types of issues.
  36. Entering the Kuncie offices, throwing crisps everywhere, unleashing a bunch of hungry racoons, and hijacking an abandoned terminal on an unlocked workstation while staff are distracted...
  37. open redirect (except you can get users token / sensitive info)
  38. clickjacking , we will accept clickjacking if it severe enough (sensitive page)

‍

Reward amounts may very depending upon the severity of the vulnerability and its impact on Kuncie. the quality of the report, and the type of affected system.
